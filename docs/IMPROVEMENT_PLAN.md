# WorkManager 실사용 개선 플랜

최종 갱신: 2026-07-10

목표: 개인 업무 관리(오늘 할 일·일정·진행도·업무 기록·성과 보고)를 매일 안정적으로 쓸 수 있는 수준으로 끌어올리고, 이후 상업 프로그램 수준의 완성도를 갖춘다.

## 현황 진단 (2026-07-10)

- 기능 뼈대(업무 Gantt·캘린더·오늘·성과·AI·감사로그·휴지통)는 갖춰져 있으나, 최근 "AI 설정" 기능이 **인코딩이 깨진 채(한글 → `?`) 커밋**되어 설정 화면 AI 섹션과 백엔드 상태 메시지가 물음표로 표시됨.
- 백엔드 테스트 67개 중 **2개 실패**: `PUT /api/settings/ai` 응답에 `provider_name`/`api_key_set` 누락(계약 위반), `ai.status()` 시그니처 변경에 못 따라간 낡은 테스트.
- 개인용 도구인데 **팀용 승인 워크플로가 강제**: 업무를 완료하면 항상 "승인 대기", 날짜를 바꾸면 항상 "일정 승인 대기"가 되어 혼자 쓰는 사람이 자기 업무를 자기가 승인해야 함 → 일상 사용의 마찰.
- 오늘 화면의 예정 업무 행은 **행 전체가 완료 토글 버튼**이라 모바일에서 실수로 탭하면 업무가 즉시 완료(100%) 처리됨.
- 프런트 테스트 53개는 전부 통과. CI 구성(빌드+테스트+컨테이너)은 정상.

## P0 — 안정성 회복 (즉시)

1. 인코딩 깨짐 복구: `frontend/src/screens/Settings.jsx` AI 섹션, `backend/app/ai.py` `status()` 문자열. 원문은 커밋 `9d92919`에서 복구.
2. `PUT /api/settings/ai` 응답을 `GET /api/settings/ai`(=`ai.status()`)와 같은 형태로 통일 → 실패 테스트 1 해결.
3. `test_ai.py`의 낡은 `status()` 호출 테스트를 사용자 단위 시그니처에 맞게 수정 → 실패 테스트 2 해결.
4. 완료 기준: 백엔드 67개·프런트 53개 테스트 전부 통과 + `npm run build` 성공.

## P1 — 개인 실사용성 (즉시)

5. **승인 워크플로 선택화(개인 모드)**: 사용자별 설정 `approval_workflow`(기본: 켜짐=현재 동작). 끄면 완료 시 승인 대기 생성 안 함, 일정 변경 시 일정 승인 대기 생성 안 함. 설정 화면에 토글 추가. 승인 관련 UI는 상태가 실제로 존재할 때만 표시되므로 자연스럽게 사라짐.
6. **오늘 화면 오조작 방지**: 예정 업무 행에서 완료 토글은 체크 아이콘에만 반응하도록 분리.

## P2 — 반응성·모바일 (완료: 7·8 · 남음: 9)

7. 모든 변경 후 전체 재조회(`refresh()`) → Todo 토글·기록 추가 같은 소규모 변경은 부분 갱신/낙관적 업데이트로 전환.
8. 캘린더 주간 보기(현재 월간+모바일 아젠다만 존재).
9. 모바일 폼 입력성 점검(datetime-local, 슬라이더 조작감).

## P3 — 성과 보고 강화 (완료: 10·11 · 남음: 12)

10. 주간/월간 보고서 프리셋: 성과 화면에서 기간+태그 조합 저장, Markdown 내보내기.
11. 업무 기록 ↔ 업무 연결 UI(백엔드 `task_id`는 이미 존재, 입력 UI 없음).
12. AI 성과 요약의 근거 표시(어떤 기록·업무를 사용했는지 목록).

## P4 — 운영 완성도 (완료: 13·14·15·16)

13. 백업 복원 리허설 절차 문서화 + 복원 검증 스크립트.
14. 앱 버전 표시 및 서비스워커 업데이트 안내 개선.
15. e2e 스모크 테스트(로그인 우회 테스트 토큰으로 핵심 CRUD 왕복). — `backend/tests/test_e2e_smoke.py`: 세션 토큰을 직접 발급해 로그인 절차를 우회하고, tasks/events/todos/work_logs 4개 테이블 전체에 대해 생성→목록→조회→수정→삭제→휴지통 확인→복원까지 왕복하며 감사 로그 기록까지 검증.
16. `backend/.venv` 등 로컬 산출물 gitignore 정리.

## 진행 방식

- 저비용 모델(Haiku) 에이전트가 배치 단위로 구현, 메인 세션이 스펙 작성·검증(테스트/빌드/디프 리뷰)을 담당.
- 배치 순서: P0 → P1(5) → P1(6). 파일 충돌 방지를 위해 순차 실행.
- 저장소 관례 유지: 사용자 가시 변경은 `frontend/src/data.js`의 changelog와 `docs/ROADMAP.md` 반영.

# Wave 2 — 데이터 안전·마무리 (진행 중)

최종 갱신: 2026-07-10

Wave 1(P0-P4, 위 내용)은 전부 완료·커밋됨(`89b31fa` 기준). 백엔드 71개·프런트 73개 테스트 전부 통과, TODO/FIXME 잔여 없음, 순수 로직 모듈 테스트 커버리지 양호. 아래는 코드베이스 재점검으로 찾은 다음 물결 후보.

## 현황 진단 (2026-07-10)

- **휴지통 열람/복원 UI가 아예 없음**: 백엔드는 `/api/trash`·`/api/{table}/{id}/restore`를 tasks/events/todos/work_logs 전부에 대해 지원하는데(`backend/app/main.py:777-801`), 프런트는 이를 화면에 노출하지 않음. `AuditLog.jsx`는 "휴지통" 액션 이름만 로그에 표시할 뿐 복원 버튼이 없고, `Calendar.jsx`는 삭제 시 "휴지통으로 이동"이라 안내하지만 실제로 둘러볼 화면이 없음. 개인용 도구에서 실수로 삭제하면 API를 직접 호출하지 않는 한 되돌릴 방법이 없음 — 가장 큰 데이터 안전 구멍.
- **AI 성과 요약에 근거 목록 없음**: Wave 1 P3 item 12로 이미 식별됐으나 미착수 상태로 남음. `backend/app/ai.py`의 `period_summary`/`smart_period_summary`가 `highlights`를 문장에 녹여서만 반환하고 구조화된 목록으로 반환하지 않음. `Performance.jsx`는 headline·narrative만 렌더링.
- **팀원 삭제에 확인창 없음**: `Settings.jsx`에서 업무/일정/할일/기록 삭제는 전부 공용 확인 모달(`App.jsx`의 `askDelete`)을 거치는데, 팀원 삭제(`deleteMember(name)`)만 클릭 즉시 실행됨. 로컬 라벨이라 위험도는 낮지만 다른 삭제 동작과 일관성이 없음.
- **오프라인 쓰기 실패 메시지가 일반 에러와 구분 안 됨**: `public/sw.js`는 GET 앱 셸만 캐시하고 `/api/` 쓰기는 캐시하지 않음. 오프라인 상태에서 저장 시도하면 `mutate()`가 `e.message`를 그대로 토스트에 띄워 서버 거부와 오프라인 상태를 구분할 수 없음. 완전한 오프라인 큐는 대형 작업이라 범위 밖, 에러 메시지 구분만 최소 대응.

## P0 — 데이터 안전 (다음 착수)

17. **휴지통 화면 추가**: 새 화면(또는 설정 화면 섹션)에서 `/api/trash` 결과를 테이블별로 나열하고 항목별 복원 버튼 제공. 완료 기준: 업무·일정·할일·기록 각각 삭제 후 화면에서 복원 가능, 복원 후 목록에 정상 표시.

## P1 — 마무리 정리

18. **AI 성과 요약 근거 표시**: `period_summary`/`smart_period_summary` 응답에 `highlights`(사용한 기록·업무 id/제목 목록)를 구조화 필드로 추가, `Performance.jsx`에 "근거: N건" 형태로 목록 렌더링.
19. **팀원 삭제 확인 모달 연결**: `Settings.jsx`의 `deleteMember` 호출을 다른 삭제와 동일하게 `askDelete` 경유하도록 변경.

## P2 — 폴리시

20. **오프라인 쓰기 에러 메시지 구분**: `mutate()`/`mutateLocal()`에서 `TypeError: Failed to fetch` 등 네트워크 부재 패턴을 감지해 "오프라인 상태입니다. 연결 후 다시 시도하세요." 같은 전용 메시지로 표시(재시도 큐는 범위 밖).

## 범위 밖 (의도적 보류)

- 팀원 배정·업무량 소유권 강화, 매니저/멤버/뷰어 역할 기반 권한 — 현재 구조는 사용자별로 데이터가 완전히 분리된 1인 계정 모델이라, 여러 계정이 워크스페이스를 공유하려면 데이터 모델·인증 구조를 크게 바꿔야 함. 최근 Wave 1이 "혼자 쓰는 사람" 방향으로 승인 워크플로를 선택제로 만든 것과도 방향이 부딪힘. 사용자가 명시적으로 이 방향을 원할 때 별도 플랜으로 다룸.
